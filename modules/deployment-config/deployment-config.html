Deployment Config

One codebase, one client per deployment — differentiated ONLY by environment. This is the short reference for the env vars that stand up a new client (BigBoss, Eternal, a US store …) without touching source: what localizes the money + calendar, what turns modules on/off, what sets the brand, and what MUST be isolated. Set these in the deployment's .env; everything else client-specific is data you seed in the admin UI. Full runbook: be-prod/DEPLOYMENT.md.

STEP 01·ORIENT

Four config seams — the whole differentiation surface

A new client is a deployment config, not a fork. Four seams cover everything: LOCALIZATION (one switch sets currency, timezone, fiscal year, chart of accounts, and tax regime together), FEATURES (which modules boot), BRANDING (invoice/notification identity), and ISOLATION (never share a database or media store between clients). If you're editing a .ts file to onboard a client, stop — it belongs in .env or the PlatformConfig admin UI. The only code change to add a whole new COUNTRY is registering its pack (a shared capability, below), not per-client work.

2

STEP 02·RESOLVE

Localization — set COUNTRY_PACK, inherit currency + timezone + fiscal year

COUNTRY_PACK is the single localization switch. It selects the ledger's chart of accounts + tax pack AND drives the deployment's base currency, report timezone, and fiscal-year start (config.locale). Set only COUNTRY_PACK and inherit all three — BD → BDT / Asia-Dhaka / July; a US pack → USD / America-* / January. The per-field overrides exist only for the rare deployment that diverges from its country's norm (a BD company keeping books in USD). The timezone is VALIDATED at boot: an invalid IANA zone (a typo, or a POSIX name like GMT+6) fails the app to start rather than silently reporting every near-midnight transaction in UTC — 6h off Dhaka, into the wrong day/month/fiscal-year.

3

STEP 03·RESOLVE

Features — turn modules on, off, or to a tier

Each module is gated by FEATURE_<ID>. A disabled feature registers ZERO routes and never boots its engine — it's not hidden, it's absent. `core` can't be disabled. FLOW_MODE separately sets warehouse depth: `simple` is basic stock; `standard` adds procurement, lots, packages, replenishment; `enterprise` adds waves, labor, traceability, and the depreciation auto-post cron. Tune these to the client's actual footprint — an online-only boutique runs FEATURE_POS=off, a 3PL runs FLOW_MODE=enterprise.

4

STEP 04·RESOLVE

Branding — the client's name, without a code edit

Backend-facing identity (invoice PDF header, notification 'from') is resolved entirely from CLIENT_* env — no source edit per client. Unset falls back to the BigBoss defaults, which double as the documentation of each var. Frontend branding (logo, theme, storefront chrome) is the Next.js app's own client.config, separate from these.

5

STEP 05·RESOLVE

Isolation — the one thing you must never get wrong

This platform is single-tenant per deployment — branches are the internal multiplicity, NOT clients, and there is no client dimension in the schema. So two clients must never share state. Give each its own database (a shared MONGO_URI would merge two companies' books) and its own media namespace (S3_MEDIA_PREFIX, or a separate bucket) and a distinct auth secret. This is a provisioning rule, not a toggle — get it right before the first request lands.

6

STEP 06·RESULT

Add a new country once — every client in it inherits it

Onboarding a NEW COUNTRY is the only code step, and it's shared across every client in that country. Add its money+calendar row to config/sections/locale.config.ts, register its chart+tax pack in resources/accounting/country-pack.ts, then set COUNTRY_PACK=<XX> in the deployment. A boot guard asserts the locale currency matches the ledger pack so the two can't drift. After that, standing up the 2nd, 3rd, Nth client in that country is pure .env — the whole point of the model.
