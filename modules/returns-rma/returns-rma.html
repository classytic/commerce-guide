Returns & RMA

A return is the sale run backwards — but it is not one reversal, it is three. When an admin confirms an RMA, a single event (order:change.confirmed) fans out to three independent handlers: goods move back, the ledger reverses, and money is refunded. The pivot is the per-line DISPOSITION: whether each returned unit re-enters the balance sheet as a sellable asset or leaves it as a loss. Follow one confirmation and watch all three resolutions fall out of that one field.

1

STEP 01·NAVIGATE

The RMA queue — returns, exchanges, claims

Every Return, Exchange and Claim across all orders lands here, filterable by type and status. A request is created as a draft / pending review — at that point nothing physical or financial has happened. It is an intent, exactly like a purchase order was on the buy side. Only CONFIRM turns it into money and stock movement.

2

STEP 02·CLICK

Raise a request — with a disposition per line

New Request lets ops create an RMA on a customer's behalf (phone, in-store, escalation). Pick Return / Exchange / Claim, then add the exact lines and quantities coming back. The decisive input is each line's DISPOSITION — restock (goods are sellable again) versus damaged / scrap / write-off (goods are dead). This one field forks the entire accounting below; it is set at creation time, not guessed later.

3

STEP 03·RESOLVE

One confirm fires THREE independent resolutions

Confirming the request emits order:change.confirmed, and three subscribers react to it independently — one for physical goods, one for the ledger, one for the money. They are decoupled on purpose: a gateway refund failure must never roll back the restock, and reversing the ledger must not wait on the courier. Three concerns, three handlers, same event.

4

STEP 04·RESOLVE

Disposition decides the INVENTORY resolution

The stock handler runs a Flow moveGroup that brings the units back from the virtual customer location. Where they land depends on disposition: restock → the DEFAULT (sellable) location, so the quant rejoins available stock and the storefront projection re-syncs; damaged / scrap → the ADJUSTMENT location, off the sellable books for good. Same return, two very different physical destinations — driven entirely by the per-line field from step 2.

5

STEP 05·RESOLVE

...and the LEDGER resolution splits on the same field

The ledger handler posts a different journal entry per line depending on disposition. RESTOCK is a COGS REVERSAL: Dr Inventory (1164) / Cr COGS (5111) — the asset comes back on the balance sheet at the exact cost layer it left on, cancelling the original cost of sale. DAMAGED is an INVENTORY LOSS: Dr Shrinkage (6711) / Cr Inventory (1164) — goods left the warehouse permanently, just not in a sale. The split matters: posting a COGS reversal for damaged goods would falsely inflate inventory and understate the true cost of running returns.

6

STEP 06·RESOLVE

Money settles separately — and skips COD

The refund handler reverses the sale's PRICE side, mirroring the sales contract: Dr Sales Revenue (4111) + Dr Output VAT (2132) — lowering both revenue and the VAT liability — Cr the SAME instrument the customer paid on (card → Gateway Clearing 1125, cash → Petty Cash 1111). It deliberately SKIPS COD orders: the courier never handed over money, so ops settle those manually. Because output VAT drops here, the return also flows into next month's Mushak 9.1. When cumulative refunds reach the order's grand total, the order transitions to 'refunded' — the sale fully unwound across goods, ledger and cash.
